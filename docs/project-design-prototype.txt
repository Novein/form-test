# 一、技术选型建议

结合你现在的项目特点：

* 纯前端
* iPad 使用
* 配置驱动表单
* 本地存储
* 多记录管理
* PDF 导出
* 先求稳定落地，不追求炫技

我建议技术栈定为：

## 推荐方案

* **React**
* **Vite**
* **TypeScript**
* **IndexedDB**
* **React Router**
* **本地 JSON 配置文件**
* **html2canvas + jsPDF** 或 **@react-pdf/renderer 二选一**

---

## 为什么这样选

### 1. React

适合表单和状态管理，后续扩展也方便。
你的项目已经不是一个简单单页，而是：

* 新建记录
* 草稿恢复
* 多页面切换
* 记录管理
* 导出预览

React 比原生 JS 更稳。

### 2. Vite

启动快，配置轻，适合纯前端项目。

### 3. TypeScript

你这个项目是配置驱动，字段会比较多。
TS 可以明显减少后续字段混乱的问题。

### 4. IndexedDB

比 localStorage 更适合保存多条记录和结构化数据。

### 5. React Router

虽然页面不多，但还是建议正规做路由。

### 6. JSON 配置文件

最符合你“抽离巡检项、结果选项、数量不确定”的需求。

---

## UI 方案建议

为了方便实用，建议：

* **不强依赖重型 UI 框架**
* 可以用少量现成组件库辅助

### 两种路线

#### 路线 A：自己写基础样式

优点：

* 轻
* 控制力强
* iPad 场景够用

#### 路线 B：用 Ant Design Mobile

优点：

* 对移动端友好
* 表单组件全
* 上手快

### 我的建议

第一版可以：

* **React + 自定义基础样式**
* 如有需要，少量引入移动端组件

因为你的页面其实比较固定，不是通用业务后台，自己控样式更合适。

# 三、页面结构设计

我建议第一版页面控制在 6 个核心页面。

---

## 1. 首页 HomePage

### 作用

作为系统入口，减少现场人员操作复杂度。

### 页面内容

展示几个大按钮：

* 新建巡检记录
* 查看记录列表
* 合并导出 PDF
* 配置说明/系统说明

### 推荐布局

iPad 上做成大卡片按钮，方便点按。

### 页面目标

让现场工人一眼知道下一步该做什么。

---

## 2. 新建记录页 NewRecordPage

### 作用

填写巡检前的基础信息。

### 字段建议

* 工人姓名
* 日期
* 班次
* 生产线
* 车间
* 巡检区域
* 巡检模板

### 页面操作

* 点击“开始填写”
* 若存在未完成草稿，可提示是否继续上次草稿

### 设计原则

这一页只负责“确定本次是谁、什么时间、什么模板”。

---

## 3. 巡检填写页 InspectionFormPage

### 作用

核心业务页，工人逐项填写巡检内容。

### 页面结构

建议按 section 分块显示，例如：

* 设备运行检查
* 工艺参数检查
* 产品质量检查
* 现场环境检查
* 补充说明

### 每个巡检项展示内容

* 项目名称
* 输入控件
* 单位
* 异常备注输入框
* 必填提示

### 页面底部操作

* 暂存草稿
* 提交记录
* 返回首页
* 清空当前填写（谨慎）

### 关键交互

* 自动保存
* 异常项自动展开备注
* 未填项高亮
* 提交前校验

---

## 4. 记录列表页 RecordListPage

### 作用

管理当前设备上的巡检记录。

### 页面内容

上半部分做筛选条件，下半部分做记录列表。

### 筛选条件建议

* 日期
* 工人
* 班次
* 生产线
* 状态：草稿 / 已提交
* 是否已导出

### 列表字段建议

* 工人姓名
* 日期
* 班次
* 生产线
* 模板名称
* 状态
* 更新时间
* 是否已导出

### 每条记录可操作

* 查看详情
* 编辑
* 删除
* 勾选导出

### 批量操作

* 批量选择
* 批量导出
* 批量删除（可选，谨慎）

---

## 5. 记录详情页 RecordDetailPage

### 作用

查看单条记录详情，也可进入编辑。

### 页面内容

* 基础信息区
* 巡检结果区
* 备注区
* 时间信息区

### 页面操作

* 编辑
* 删除
* 返回列表

### 是否单独保留这个页面

建议保留。
因为列表页不适合承载过多详情内容。

---

## 6. 导出预览页 ExportPreviewPage

### 作用

在真正生成 PDF 前，先把选中的记录汇总展示出来。

### 页面内容

* PDF 标题
* 巡检日期
* 生产线 / 班次
* 选中的记录数
* 每位工人的巡检结果
* 异常项重点展示
* 导出时间

### 页面操作

* 生成 PDF
* 返回重新选择
* 移除某一条记录

### 为什么建议加预览页

因为这是业务闭环里很重要的一步：

* 防止选错记录
* 防止导出前才发现信息不全
* 便于确认合并结果

```text

```

# 五、页面交互流程设计

下面开始讲得更细一点，进入“怎么操作”的层面。

---

## 1. 首页交互

### 按钮

* 新建巡检记录
* 查看记录
* 合并导出 PDF

### 建议

首页不放复杂内容，不放统计图，不放太多文字。

---

## 2. 新建记录页交互

### 默认行为

* 日期默认当天
* 班次可默认最近一次使用值
* 生产线可默认最近一次使用值
* 工人从配置名单选择

### 进入填写页前校验

至少校验：

* 工人必选
* 日期必填
* 班次必选
* 生产线必选
* 模板必选

### 草稿恢复逻辑

若检测到相同工人 + 相同日期 + 相同班次 + 相同产线 下存在草稿：

* 提示继续草稿
* 或新建一条记录

建议第一版做一个简单提示即可。

---

## 3. 巡检填写页交互

### 表单展示逻辑

按模板 sections 顺序展示。
每个 section 用卡片块展示。

### 字段级交互规则

#### radio 类型

* 点击即选中
* 若选中“异常”，自动显示备注框

#### number 类型

* iPad 上弹数字键盘
* 支持输入范围校验

#### text / textarea

* 普通文本输入

### 自动保存规则

建议：

* 输入变化后延迟 1 到 2 秒自动保存
* 页面切换前再强制保存一次

### 提交规则

点击提交时：

1. 执行必填校验
2. 执行异常备注校验
3. 提示确认提交
4. 提交后状态变更为 `submitted`

---

## 4. 记录列表页交互

### 筛选交互

建议采用“轻筛选”：

* 上方一排筛选控件
* 下方立即刷新列表

### 列表交互

每条记录左侧可勾选，右侧放操作按钮：

* 查看
* 编辑
* 删除

### 批量导出逻辑

* 至少选择 1 条已提交记录
* 不允许选择草稿参与导出
* 若记录模板不一致，可提示暂不支持混合导出

---

## 5. 导出预览页交互

### 页面展示重点

建议优先突出：

* 基础信息
* 每份记录的工人信息
* 异常项汇总
* 全量巡检结果

### 生成 PDF 前校验

* 记录不能为空
* 必须都是已提交状态
* 建议同模板
* 建议同日期或同班次，若不一致给出提示

### 导出成功后

可选做法：

* 标记所选记录 `exported = true`
* 记录 `exportedAt`

第一版建议这样做，方便后续管理。

# 六、字段流转方案

这个部分非常重要，相当于系统里的“数据生命线”。

---

## 1. 新建时生成哪些字段

当用户在“新建记录页”点击开始填写时，生成一条 `InspectionRecord` 草稿数据：

```text
基础信息
+ 空 answers
+ status = draft
+ createdAt
+ updatedAt
+ exported = false
```

---

## 2. 填写时字段怎么流转

### 基础信息字段

这些字段一般在创建后就固定：

* workerId
* workerName
* inspectionDate
* shiftId
* productionLineId
* templateId

### 表单答案字段

随着填写不断更新：

* answers[itemId].value
* answers[itemId].remark
* updatedAt

### 自动保存

每次更新写入 IndexedDB。

---

## 3. 提交时字段怎么变化

点击提交后：

```text
status: draft → submitted
submittedAt: 当前时间
updatedAt: 当前时间
```

如果允许编辑已提交记录，则后续修改只更新：

* answers
* updatedAt

一般不改 `submittedAt`，除非你要支持“重新提交”。

---

## 4. 导出时字段怎么变化

点击导出成功后，对所选记录更新：

```text
exported = true
exportedAt = 当前时间
updatedAt = 当前时间
```

这样列表页就能看出哪些记录已经导出过。

# 七、表单渲染规则设计

这一部分是配置驱动的关键。

---

## 渲染逻辑建议

### 1. 读取模板

根据 `templateId` 找到模板对象。

### 2. 遍历 sections

按 `sortOrder` 排序后展示。

### 3. 遍历 items

根据 `type` 渲染不同组件。

---

## 每种类型对应组件建议

### radio

组件：`RadioField`

输入：

* label
* options
* value
* required
* allowRemark
* remarkRequiredWhen

### number

组件：`NumberField`

输入：

* label
* unit
* min
* max
* placeholder

### text

组件：`TextField`

### textarea

组件：`TextareaField`

---

## 字段校验规则建议

### 通用校验

* required = true 时必须有值

### radio 额外校验

* 若配置了 `remarkRequiredWhen`
* 且当前值命中条件
* 则 remark 必填

### number 额外校验

* 校验 min / max
* 校验是否为数字

# 九、状态管理建议

第一版不要上太重的状态管理。
建议：

* 页面局部状态：`useState`
* 表单状态：`useReducer`
* 全局少量配置：`Context`

---

## 为什么建议表单页用 useReducer

因为巡检填写页状态会比较复杂：

* 基础信息
* answers
* 保存状态
* 校验状态
* 自动保存状态

`useReducer` 会比一堆 `useState` 更清晰。
